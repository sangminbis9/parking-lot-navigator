# Readiness Review

배포 준비 점검 보고서를 **현재 저장소 상태에 맞게 재검토·갱신**하는 스킬이다. 보고서를 처음부터 쓰는 게 아니라, 이미 존재하는 보고서를 기준점으로 삼아 (1) 그동안 해결되거나 바뀐 항목을 실제 근거로 확인해 고치고, (2) 새로 생긴 차단 요소·권장 사항을 추가한다.

대상 파일 기본값: `docs/release/deployment-readiness-report.md`
(다른 경로가 지정되거나 그 파일이 없으면 사용자에게 확인한다.)

## 핵심 원칙: 추정이 아니라 근거로 판단한다

이 스킬에서 가장 위험한 실패는 **확인 없이 "해결됨"으로 바꾸는 것**이다. 보고서를 읽는 사람은 이 표를 보고 출시 여부를 결정한다. 근거 없는 🟢 OK는 차단 요소를 통과시키는 거짓 신호가 된다.

그래서 모든 상태 변경은 추적 가능한 근거에서 나와야 한다:

- 파일/디렉터리 존재 여부 (예: `ios-app/.../PrivacyInfo.xcprivacy`가 실제로 생겼는가)
- 명령 실행 결과 (typecheck/test 통과, lint 결과)
- 코드/설정 내용 (예: Toss 키가 `test_`→`live_`로 바뀌었는가, CORS가 좁혀졌는가)
- 지난 점검 이후의 git 변경 (`git log <지난커밋>..HEAD`)
- 새로 추가된 문서 (예: `docs/legal/terms.md`가 생겼는가)

근거를 찾지 못하면 상태를 바꾸지 말고 그대로 둔다. 저장소만으로 확인 불가능한 항목(사업자등록, 외부 URL 호스팅 완료, App Store Connect 입력 등)은 임의로 닫지 말고, 필요하면 표의 상태에 "코드로 확인 불가 — 사용자 확인 필요"라고 명시한다.

## 허용/금지 작업

이 스킬은 **읽기와 비파괴 명령만** 한다. 보고서 파일 외에는 아무것도 바꾸지 않는다.

허용: 파일 읽기, `rg`/`find`로 검색, `git status`/`git log`/`git diff`, `pnpm -C worker-backend typecheck`, `pnpm --filter @parking/backend test`, 파일 존재 확인 등 상태를 바꾸지 않는 명령.

금지: 코드/설정 수정, 마이그레이션 실행, `wrangler deploy`, git commit/push, 외부로 데이터 전송. 이 스킬의 산출물은 오직 갱신된 보고서 마크다운 하나다.

## 작업 흐름

### 1. 보고서 읽기와 기준점 파악
- 대상 보고서를 **끝까지** 읽는다. 추적 중인 모든 항목·심각도·섹션 구조·표 포맷을 머릿속에 넣는다.
- 헤더에서 지난 점검의 **작성일**과 **대상 커밋**을 확인한다. 이게 "그동안 무엇이 바뀌었나"의 기준선이다.

### 2. 변경 근거 수집
먼저 번들된 스크립트로 공통 신호를 모은다:

```bash
bash .claude/skills/readiness-review/scripts/gather-evidence.sh [지난_대상_커밋_or_날짜]
```

이 스크립트는 현재 커밋, 지난 점검 이후 git 로그, 주요 릴리스 산출물 존재 여부, D1 마이그레이션 목록 등을 출력한다. best-effort이므로 일부 실패해도 무시하고 진행한다. 스크립트 출력은 출발점일 뿐이고, 항목별 검증은 직접 한다.

### 3. 정밀 재검사 (보고서 섹션 = 검사 범위)
보고서의 각 섹션을 순서대로 훑으며, 그 섹션이 다루는 영역을 실제로 들여다본다. 어디를 봐야 하는지는 `references/inspection-map.md` 참고. 보고서에 있는 섹션이 곧 점검 범위이므로, 섹션을 임의로 지우거나 통째로 재구성하지 않는다.

### 4. 추적 항목 대조
각 기존 항목에 대해:
- **해결됨**: 근거로 확인되면 상태를 🟢로 바꾸고, "권장 조치" 칸에 무엇으로 해결됐는지(예: "PrivacyInfo.xcprivacy 추가됨 @commit")를 한 줄 남긴다. 행을 지우지 말 것 — 해결 이력도 정보다.
- **상태 변화**: 심각도나 상황이 바뀌었으면 "현재 상태"/"우선순위" 칸을 갱신한다.
- **그대로**: 근거가 없으면 손대지 않는다.

### 5. 새 항목 발견
재검사 중 보고서에 없는 차단 요소·위험·개선 기회를 찾으면, 가장 알맞은 섹션에 같은 표 포맷으로 행을 추가한다. 어느 섹션에도 안 맞으면 적절한 새 항목 위치를 정하되, 새 최상위 섹션 신설은 정말 필요할 때만 한다. 추가 항목도 근거 기반이어야 한다 — "있으면 좋을 것 같은" 추측성 항목을 남발하지 않는다.

### 6. 요약·P0·로드맵·헤더 동기화
표를 고쳤으면 보고서 상단/하단의 파생 정보도 일관되게 맞춘다:
- 헤더의 **작성일**(오늘 날짜)과 **대상 브랜치/커밋**(현재 HEAD)을 갱신한다.
- **Executive Summary**와 **P0 정리** 섹션을, 갱신된 표의 실제 🔴 항목과 일치시킨다. P0가 해결됐으면 요약에서도 빼고, 새 P0가 생겼으면 넣는다.
- **로드맵**에 영향이 있으면 반영한다.

### 7. 변경 로그 남기기
보고서 상단(목차 근처)에 `## 변경 로그` 섹션이 없으면 만들고, 이번 실행을 한 줄로 기록한다:

```
## 변경 로그
- 2026-05-31 (@<commit>): 해결 N건(…), 신규 N건(…), 상태변경 N건(…)
```

이렇게 하면 사용자가 매 실행마다 무엇이 달라졌는지 보고서만 봐도 안다.

### 8. 사용자에게 보고
보고서 갱신 후, 채팅에는 짧게 요약한다: 해결된 항목, 새로 추가한 항목, 상태가 바뀐 항목, 그리고 **저장소만으로 확인 불가해서 사용자 확인이 필요한 항목**. 마지막 항목을 빠뜨리지 말 것 — 자동 점검의 사각지대를 사용자가 알아야 한다.

## 포맷 규칙

기존 보고서의 스타일을 그대로 따른다. 새로 발명하지 않는다.

- 심각도 마커는 보고서가 쓰는 표기를 그대로: 🔴 P0(차단) · 🟡 P1(출시 전) · 🟠 P2(출시 후) · 🟢 OK/보류.
- 표 컬럼 구성(항목 / 현재 상태 / 우선순위 / 권장 조치)을 유지한다.
- 한국어로 쓴다.
- 기존 항목의 표현을 불필요하게 다시 쓰지 않는다. 바뀐 부분만 surgical하게 고친다. diff가 작아야 사용자가 무엇이 달라졌는지 검토하기 쉽다.

## 정직성 규칙

- 확인 못 한 걸 확인한 척하지 않는다. typecheck를 안 돌렸으면 그 항목 상태를 "통과"로 바꾸지 않는다.
- 저장소 밖 사실(외부 URL 게시 여부, 법무 진행, 스토어 입력)은 코드로 모른다. 추측해서 닫지 말고 "사용자 확인 필요"로 둔다.
- 명령이 실패하면 숨기지 말고, 해당 항목 상태에 "검증 시도했으나 실패: <이유>"를 남긴다.
